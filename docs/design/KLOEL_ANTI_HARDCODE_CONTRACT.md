# Kloel Anti-Hardcode Contract

## Status

- Mandatory.
- Applies to backend, frontend, worker, prompts, metadata persistence,
  streaming, and any AI-facing surface.

## Absolute Rule

If text appears in the product as if it came from Kloel's intelligence,
reasoning, or live activity, that text must come from the model or it must not
exist.

There is no middle ground.

## What May Be Hardcoded

- Static interface labels:
  - `Copiar`
  - `Gostei`
  - `Não gostei`
  - `Tentar novamente`
  - `Enviar`
  - `Cancelar`
  - `Nova conversa`
- Explicit system errors:
  - connectivity failures
  - timeout failures
  - server errors
- Non-model placeholders for controls:
  - composer placeholder
  - button labels
  - navigation labels

## What May Not Be Hardcoded

- “Thinking” prose such as:
  - “Entendendo sua pergunta...”
  - “Redigindo a resposta...”
  - “Processando sua solicitação...”
  - “Executando a ferramenta necessária...”
- Trace summaries formed by concatenating pipeline labels.
- User-facing descriptions derived from tool names like `search_web` ,
  `create_payment_link` , or `update_lead_status` .
- Fake reasoning, fake internal monologue, fake operational storytelling, or
  fake activity cards.
- Greeting/capability text presented as if it were dynamically reasoned by the
  model when it was written in the code.

## Processing State Contract

- If the provider/model does not offer native human-readable reasoning:
  - show only the branded animation and an elapsed timer
  - optionally show a slow-response warning and a cancel action
  - do not invent explanatory text
- If the provider/model does offer reasoning:
  - show only the provider/model output
  - do not wrap the provider gap with fallback fake cognition

## Tool Call Contract

- Internal tool names are not product copy.
- Tool-call UI may exist for operator/developer consoles, but must not
  masquerade as AI reasoning in customer/product chat.
- On the customer-facing Kloel chat surface:
  - technical tool names are hidden by default
  - no regex-based prettification of tool names is treated as model speech

## Summary Contract

- A summary of what Kloel did may appear only when:
  - it is generated by the model
  - or it is clearly marked as a system/runtime state instead of model cognition
- Runtime/system state copy must never pretend to be the model’s internal
  thought.

## Model Routing Contract

- Substantive Kloel chat turns must use the stronger GPT-5.4 “brain” path, not
  the cheapest nano path.
- Trivial turns may use the lighter path when quality and identity are not
  compromised.
- If the system cannot provide the promised deep reasoning experience, it must
  fail honestly rather than simulate it.

## Workspace Context Contract

- Real workspace/business context must come from real data.
- Do not hardcode placeholders like `"Workspace"` when the real workspace name
  is available.

## Conversation Title Contract

- Greeting-only openings must not generate junk permanent titles.
- Keep `Nova conversa` until a substantive message exists.
- Generated titles must come from meaningful user intent, not filler greetings.

## Grep Enforcement

The following pattern is a guardrail. If it finds matches in production chat
code, the contract is probably being violated:

```bash
grep -rn "Entendendo sua\\|Redigindo a resposta\\|Processando sua\\|Executando a ferramenta\\|Integrando o resultado\\|Consolidando os resultados\\|Reprocessando esta\\|Resposta pronta\\|reunindo o contexto\\|Avaliando se precisa" backend/src frontend/src
```

This is not the only enforcement mechanism, but it is a fast smoke test for
deceptive pseudo-intelligence.

## Shipping Rule

- No PR that reintroduces pseudo-thinking, fake cognition, fake tool narration,
  or model-impersonating hardcode is acceptable.
