## Context

KLOEL's path to its first production deployment crosses 6 phases, 4 tiers of
module maturity, 50+ PULSE parsers, an escalating Codacy ratchet, ~40 provider
integrations, a Stripe migration, and a WhatsApp dual-provider architecture.
Each day brings new findings, new CI outcomes, coverage drift, and dependency
changes. The surface area of production-readiness information is ~15 distinct
data sources changing independently.

The question was: how to surface this state to **two radically different
consumers** — Daniel (visual, spatial, needs at-a-glance triage) and
Claude/OpenCode (textual, needs low-token state reads via MCP tools) — from a
**single source of truth** that cannot drift.

### Pre-ADR state

Before this decision, KLOEL had four separate mechanisms for tracking production
readiness:

1. **PULSE** (`scripts/pulse/`) — a static auditing framework with 50+ parsers
   producing `PULSE_REPORT.md`. Every parser classifies files and emits findings.
   PULSE is read-only on the codebase; it has no persistence, no timeline, no
   visual layer.

2. **Obsidian Mirror Daemon** (`scripts/obsidian-mirror-daemon*.mjs`) — a
   constitution-locked daemon that mirrors the git working tree into an Obsidian
   vault under `<vault>/Kloel/99 - Espelho do Codigo/`. Each file is a note with
   frontmatter tags (`graph/surface-*`, `graph/risk-*`, `graph/governance`,
   `findings/severity-*`). The daemon enforces a strict scan-and-compare loop; it
   never mutates code, only mirrors it.

3. **Findings sidecar pipeline** — a set of emitters
   (`scripts/orchestration/findings-watch.mjs`,
   `scripts/orchestration/severity-tags-emitter.mjs`) that write per-file sidecar
   JSON (`<file>.findings.json`) into the mirror vault next to each node. These
   sidecars carry PULSE findings with severity, parser name, and line evidence.

4. **Graph lens** (`scripts/orchestration/extend-graph-lens.mjs`) — a
   non-protected script that injects `colorGroups` into the Obsidian Graph
   view's JSON, colourizing nodes by severity, surface, and risk category.

5. **Implicit human state** — Daniel carried state in his head: which modules are
   closest to ship, which blockers are truly blocking, which providers are
   healthy today, what changed since yesterday. This state was not queryable,
   versioned, or visible to AI agents.

The gap: these systems produce **raw signals** but no **synthesized
state-machine surface**. PULSE has findings. The mirror has tags. The sidecars
have evidence. The lens has colors. But nobody — human or agent — could answer
"what are the 3 next tasks?" without reading 5+ files and computing rank
manually.

### The architectural tension

The fork in the road was:

| Option A                                              | Option B                                        |
| ----------------------------------------------------- | ----------------------------------------------- |
| Static `PRODUCTION_SCOREBOARD.md` generated by a cron | Obsidian vault as the **live HUD surface**      |
| Daniel reads markdown; Claude reads via filesystem    | Daniel sees Graph/Canvas/Bases; Claude uses MCP |
| Separate artifact; no integration with mirror/daemon  | Reads sidecars directly from vault nodes        |

Option A is simple but creates **two sources of truth** (the mirror vault for
code + a separate scoreboard for state), guaranteeing drift. Option B makes the
vault the **unified surface** — one data layer, two consumers, zero drift.

### What Obsidian 2026 brings that didn't exist before

Obsidian in May 2026 offers capabilities that make it a viable HUD infrastructure
rather than just a note-taking tool:

- **Bases** (native core plugin, GA since May 2025) — DB views of notes-as-rows
  with typed Properties, filter/sort/group, bulk-edit inline. Renders instantly
  on 50k+ notes.
- **Properties** (typed YAML frontmatter) — vault-wide typing. `tier` is always
  int, `phase` always int, `coverage` always float. Queryable by Bases, Dataview,
  MCP tools, and Graph colorGroups.
- **Local REST API** plugin (`coddingtonbear/obsidian-local-rest-api`) —
  HTTPS+API-key CRUD on files, frontmatter, tags. The doorway for AI agents.
- **MCP servers** — 5 mature options (`cyanheads/obsidian-mcp-server@3.1.1`,
  `ToKiDoO/mcp-obsidian-advanced`, etc.). Expose vault as MCP tools to Claude
  Code, OpenCode, ChatGPT, Cursor.
- **engraph** (`devwhodevs/engraph`) — local knowledge graph for AI agents.
  Hybrid search (semantic + FTS5 + wikilink graph + temporal + LLM rerank). 25
  MCP tools + 26 REST endpoints. SQLite-backed. Zero cloud.
- **Canvas** — whiteboard with embedded notes/images/URLs. Live mind maps,
  embedded Kanban from tasks, embedded charts.
- **Excalidraw** with JS scripting API — hand-drawn living architecture diagrams.
- **Obsidian Git** plugin — auto-commit vault hourly. Full project-state
  time-machine with daily snapshots.
- **Mermaid** (native) — Gantt, flowchart, sequence, class diagrams auto-
  generated from DataviewJS.

Without these capabilities, Obsidian would be a note-taking tool with graph
gimmickry. With them, it is a **programmable state-machine surface** with typed
properties, DB views, agent MCP doorway, and time-travel.

## Decision

The Obsidian vault at `~/Documents/KLOEL/Kloel/` is adopted as the **single
source of truth** for KLOEL's path-to-production state. It becomes the
**Production HUD** — the state-machine surface consumed by Daniel (visual:
Graph, Canvas, Bases) and Claude/OpenCode (textual: MCP tools, hub notes)
without drift.

### Architecture

The HUD rests on four layers:

```
Layer 1 — Constitution-locked daemon (read-only, governance surface)
├── scripts/obsidian-mirror-daemon.mjs          ← mirror scan loop
├── scripts/obsidian-mirror-daemon-content.mjs   ← content extraction
├── scripts/obsidian-mirror-daemon-indexes.mjs   ← index generation
├── scripts/obsidian-mirror-daemon-constants.mjs ← tag namespaces
└── scripts/obsidian-mirror-daemon-utils.mjs     ← shared helpers

Layer 2 — Non-locked sidecar emitters (writable by orchestration)
├── scripts/orchestration/tier-tags-emitter.mjs        → kloel/tier-N
├── scripts/orchestration/phase-tags-emitter.mjs        → kloel/phase-N
├── scripts/orchestration/coverage-sidecar-emitter.mjs  → coverage/below-threshold
├── scripts/orchestration/ci-state-emitter.mjs          → ci/failing | ci/passing
├── scripts/orchestration/provider-state-emitter.mjs     → provider/* tags
├── scripts/orchestration/severity-tags-emitter.mjs      → findings/severity-*
└── scripts/orchestration/findings-watch.mjs             → findings pipeline trigger

Layer 3 — Auto-generated hub notes (overwritten on every refresh)
├── <vault>/Kloel/00-HUD/00-NEXT.md          ← top-3 tasks (Claude reads first)
├── <vault>/Kloel/00-HUD/00-BLOCKERS.md      ← full ranked queue, top 50
├── <vault>/Kloel/00-HUD/00-DAG.md           ← phase progress + module status
├── <vault>/Kloel/00-HUD/00-PROVIDERS.md     ← Stripe/Meta/WAHA health
├── <vault>/Kloel/00-HUD/00-REGRESSIONS.md   ← what worsened since last snapshot
├── <vault>/Kloel/00-HUD/00-HUD-README.md    ← map + last-refresh timestamp
└── <vault>/Kloel/00-HUD/snapshots/YYYY-MM-DD.md  ← daily state archive

Layer 4 — AI-agent doorway (MCP + REST)
├── cyanheads/obsidian-mcp-server@3.1.1       ← primary MCP bridge
├── engraph (devwhodevs)                        ← local KG + hybrid search
├── Claude Code MCP config                      ← auto-loads vault tools
└── OpenCode subagent dispatch                  ← reads hub notes as first action
```

### Decision D1 — Vault as single source of truth for production state

Every dimension of production readiness lives as a queryable property in the
vault:

| Dimension       | Property/Tag                                             | Source                                   |
| --------------- | -------------------------------------------------------- | ---------------------------------------- |
| Module tier     | `kloel/tier-1` … `kloel/tier-4`                          | PULSE functional coverage                |
| Phase           | `kloel/phase-0` … `kloel/phase-6`                        | CLAUDE.md DAG + PULSE module mapping     |
| Coverage        | `coverage/below-threshold`                               | Jest `lcov.info` / `coverage-final.json` |
| CI status       | `ci/failing`, `ci/passing`                               | `gh run list --json` (latest 10)         |
| Provider health | `provider/healthy`, `provider/degraded`, `provider/down` | PULSE provider sections + CI logs        |
| Findings        | `findings/severity-*`                                    | PULSE parsers                            |
| Surface         | `graph/surface-*`                                        | Mirror daemon (constitution-locked)      |
| Risk            | `graph/risk-*`                                           | Mirror daemon (constitution-locked)      |
| Governance      | `graph/governance`                                       | Mirror daemon (constitution-locked)      |

Daniel opens the Graph view and sees nodes coloured by tier (red → green), phase
(spectrum), and severity (intensity). Claude calls
`obsidian:read("00-HUD/00-NEXT.md")` via MCP and gets the top-3 tasks in ~80
tokens instead of ~1500 tokens of filesystem reads.

### Decision D2 — Hub notes as the canonical query surface

Six auto-generated `.md` files in `<vault>/Kloel/00-HUD/` are the canonical
answers to the six operational questions:

| Question                                | Hub                 | Consumers          |
| --------------------------------------- | ------------------- | ------------------ |
| What do I work on next?                 | `00-NEXT.md`        | Claude, Daniel     |
| What's blocking us, ranked by urgency?  | `00-BLOCKERS.md`    | Daniel, Claude     |
| Where are we across all phases/modules? | `00-DAG.md`         | Daniel             |
| Are Stripe/Meta/WAHA healthy?           | `00-PROVIDERS.md`   | Daniel, Claude     |
| What got worse since the last snapshot? | `00-REGRESSIONS.md` | Daniel             |
| How do I read this HUD?                 | `00-HUD-README.md`  | Daniel, onboarding |

All hub notes carry the header marker `<!-- AUTO-GENERATED — do not edit -->`
and are regenerated from scratch on every refresh by the hubs-generator
(`scripts/orchestration/hubs-generator.mjs`). This ensures zero accumulated
drift — every refresh is a full recompute from current sidecars, not an
incremental patch.

### Decision D3 — Blocker-rank algorithm as canonical prioritization engine

A weighted scoring function implemented in
`scripts/orchestration/blocker-rank.mjs` ranks every file with open findings:

```
score(file) = tier_weight × phase_priority × user_impact × (1 / effort_estimate)

tier_weight:
  tier-1 (≥80% functional, last-mile blockers) = 4.0
  tier-2 (partial)                              = 3.0
  tier-3 (façade)                               = 2.0
  tier-4 (shell)                                = 1.0

phase_priority:
  FASE 0  = 10
  FASE 1  = 8
  FASE 2  = 6
  FASE 3  = 4
  FASE 4  = 2
  FASE 5  = 1
  FASE 6  = 1

user_impact (derived from PULSE route/service tracing):
  hits user route or API endpoint = 5
  hits service consumed by route = 4
  financial / auth / messaging    = 5 (override)
  dev infra only                  = 1

effort_estimate = max(0.5, LOC/100 + open_findings_count/3 + complexity/15)
 open_findings_count is severity-weighted:
   critical = 4, high = 2, medium = 1, low = 0.5

Tiebreaker: prefer files in modules already > 60% complete (closer to ship).
```

`00-NEXT.md` shows the top 3 with path, line, reason (top finding), effort,
phase, tier, and evidence link. `00-BLOCKERS.md` shows the full ranked queue
(top 50).

### Decision D4 — Sidecar schema mandate

All metadata that is not a code mirror node is stored in **non-protected sidecar
JSON files** alongside mirror nodes in the `_source/` directory. Sidecar schemas
are versioned as `kloel.<kind>.v1`:

```json
// <file>.tier.json
{ "schema": "kloel.tier.v1", "tier": 1|2|3|4, "evidence": [...], "computedAt": "ISO8601" }

// <file>.phase.json
{ "schema": "kloel.phase.v1", "phase": 0-6, "module": "string", "evidence": [...] }

// <file>.coverage.json
{ "schema": "kloel.coverage.v1",
  "lines": { "covered": int, "total": int, "pct": float },
  "branches": { ... }, "lastRun": "ISO8601", "source": "lcov|jest" }

// ci-state.json (global, in vault root)
{ "schema": "kloel.ci.v1", "lastRun": "...", "runs": [...] }

// provider-state.json (global, in vault root)
{ "schema": "kloel.provider.v1", "providers": [...] }
```

Sidecars are written atomically (tmp file + rename). Emitters skip writes when
the computed value hasn't changed (idempotent). This prevents unnecessary Git
churn and Obsidian re-indexing.

### Decision D5 — MCP as the primary Claude/OpenCode doorway

Claude and OpenCode subagents access the vault exclusively via MCP tools, not
filesystem reads. The `cyanheads/obsidian-mcp-server@3.1.1` server is the
primary bridge, with `engraph` as the secondary semantic layer.

Operations Claude performs via MCP:

- `obsidian:read("00-HUD/00-NEXT.md")` — first action of every session (~80 tokens)
- `obsidian:search("stripe webhook")` — search across vault (~200 tokens)
- `obsidian:dataview("LIST FROM #kloel/tier-1 SORT score DESC LIMIT 3")` — ranked query (~150 tokens)
- `obsidian:update_property("path", "tier", 2)` — mark file as tier-promoted
- `obsidian:search_in_file("00-BLOCKERS.md", "stripe")` — targeted query

The token economy gain is estimated ~12x over filesystem reads for state queries
(see Consequences below).

### Decision D7 — Non-protection of orchestration surface

All Layer 2 sidecar emitters, the blocker-rank script, the hubs-generator, the
graph-lens extension, and the HUD orchestrator are **non-protected**. They can
be authored, edited, and committed by AI subagents without governance gate. This
is deliberate — the orchestration surface is the **execution layer**, not the
**constitution layer**.

The constitution-locked surface remains only:

- All `scripts/obsidian-mirror-daemon*.mjs` files (read-only to AI)
- `scripts/pulse/no-hardcoded-reality-audit.ts` (read-only to AI)
- All governance files listed in `ops/protected-governance-files.json`

## Consequences

### Positive

#### C1 — Single source of truth eliminates drift

All production-readiness state lives in one vault. Daniel sees the same data as
Claude. When an emitter updates a sidecar, the Graph re-colours, Bases
re-queries, and hub notes regenerate — all from the same write. There is no
"scoreboard says X but PULSE says Y" class of bug because the scoreboard IS the
vault, which reads from the sidecars, which read from PULSE.

#### C2 — ~12x token economy for AI agent state reads

| Operation                                | Filesystem read                                                                 | MCP query                                                                                      |
| ---------------------------------------- | ------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| "What's the next task?"                  | Read `00-NEXT.md` via Read tool: ~1500 tokens (full file + surrounding context) | `obsidian:read("00-NEXT.md")`: ~80 tokens                                                      |
| "Is Stripe healthy?"                     | Grep `provider-state.json` + read file: ~2000 tokens                            | `obsidian:properties("provider/healthy")`: ~100 tokens                                         |
| "What's blocking FASE 2?"                | Read `00-BLOCKERS.md` + `00-DAG.md` + `PULSE_REPORT.md`: ~8000 tokens           | `obsidian:dataview("FROM #kloel/tier-1 WHERE phase = 2 SORT score DESC LIMIT 5")`: ~200 tokens |
| "Show me tier-1 files with low coverage" | Grep coverage sidecars + filter tier sidecars: ~5000 tokens                     | `obsidian:dataview("FROM #kloel/tier-1 AND #coverage/below-threshold"): ~200 tokens`           |

Estimated aggregate: **~12x token savings** for state queries. Over hundreds of
Claude sessions, this compounds significantly in both cost and latency.

#### C3 — Dual-consumer architecture (Daniel visual + Claude textual)

The same data serves two orthogonal consumption patterns:

- **Daniel**: opens Obsidian → Graph view shows tier/phase/severity colour map
  → Canvas Production Wall shows embedded hub notes → Bases table lets him
  filter/sort/group blockers → Mermaid Gantt shows phase progress → Excalidraw
  shows living architecture. All visual, spatial, at-a-glance.

- **Claude/OpenCode**: starts session → `obsidian:read("00-HUD/00-NEXT.md")` as
  first tool call (~80 tokens) → decides what to work on → `obsidian:search()`
  to find evidence → `obsidian:update_property()` to mark progress. All textual,
  low-token, queryable.

No translation layer. No "Daniel sees one thing, the agent sees another."

#### C4 — Karpathy LLM Wiki pattern enables agent-maintained knowledge

The vault adapts Andrej Karpathy's LLM Wiki pattern for KLOEL:

```
Kloel/
├── 99 - Espelho do Codigo/  ← daemon-locked mirror (ingest)
│   └── _source/              ← code mirror + sidecars (raw evidence)
├── 00-HUD/                   ← auto-generated state hub (output)
├── ADRs/                     ← architecture decisions (wiki: processed)
├── Plans/                    ← implementation plans (wiki: synthesized)
├── Runbooks/                 ← operational procedures (wiki: agent-maintained)
└── _meta/
    ├── taxonomy.md           ← canonical tag hierarchy
    ├── claude-instructions.md ← what Claude MUST read on session start
    └── opencode-instructions.md ← what OpenCode subagents MUST honor
```

Karpathy operations adapted:

- `/ingest-source` — pull external evidence (PR, Sentry issue, Stripe event) → vault note
- `/refresh-hud` — regenerate 00-HUD/\* from current sidecars
- `/lint-vault` — find broken links, orphan notes, contradictions
- `/snapshot-daily` — archive today's HUD state
- `/blocker-rank` — recompute ranking
- `/next-move` — Claude reads `_meta/claude-instructions.md` → `00-NEXT.md` → decides

This makes the vault a **living document maintained by AI agents** rather than a
human-maintained wiki that inevitably decays.

#### C5 — Living architecture diagrams via Excalidraw

Instead of committing static `.png` or `.drawio` files that decay within weeks,
the vault hosts Excalidraw diagrams with JS scripting API. Architecture
diagrams can be regenerated from current code state by agents. The architecture
is always live, always current.

#### C6 — Time-machine via Obsidian Git auto-commit

The Obsidian Git plugin auto-commits the vault hourly. Daily snapshots in
`<vault>/Kloel/00-HUD/snapshots/YYYY-MM-DD.md` create a time-series of
production readiness. Answering "when did FASE 2 coverage drop below 80%?" is a
`git log --grep="coverage"` away. The vault is its own forensic record.

#### C7 — Mobile triage via vault sync

Daniel can check KLOEL's production readiness from his phone via vault sync
(Obsidian Sync or self-hosted LiveSync). Opening the mobile Obsidian app →
00-HUD/00-NEXT.md shows the top-3 tasks. No custom mobile app needed.

#### C8 — Extensible without changing the daemon

New emitters (new tag namespaces, new sidecar schemas, new hub notes) extend the
orchestration surface without touching the constitution-locked mirror daemon.
The daemon's job is "mirror code → vault." The orchestration layer's job is
"enrich vault → hub notes." These are cleanly separated.

#### C9 — Graph view as universal visual debugger

The Obsidian Graph view with colourGroups becomes a universal visual debugger
for KLOEL health:

- Red cluster? Critical-severity findings.
- Yellow cluster? `coverage/below-threshold` files.
- Bright-red nodes? `ci/failing` anchor files.
- Black node? `provider/down` — something is broken _right now_.
- Green gradient? Tier-1 files approaching ship-readiness.

A single glance tells Daniel whether the project is converging toward production
or diverging.

### Negative / Cost

#### N1 — Bootstrap cost: ~30 hours of OpenCode subagent work

Implementing the full HUD across all waves (sidecar emitters, synthesis,
orchestrator, lens extension, MCP wiring, plugin installation, theme) is
estimated at ~30 hours of parallel OpenCode V4 Pro subagent time. This is a
one-time cost amortized over the entire production path.

#### N2 — Vault adds ~2 GB disk footprint

The mirror daemon creates one markdown note per source file in the git
working tree. With ~900 tracked source files + sidecar JSON + hub notes +
Obsidian cache/indexes, the vault is expected to consume approximately 2 GB of
disk. The daemon also re-scans the entire working tree on each cycle (currently
every 60 seconds when running in `--watch` mode), consuming CPU.

#### N3 — Ongoing infrastructure (daemon, watchdog, emitters)

Three daemon processes must remain healthy:

1. **Obsidian Mirror Daemon** — must be running to reflect code changes into the
   vault. If it stops, vault state diverges from code.
2. **HUD Orchestrator** (`scripts/orchestration/hud-orchestrator.mjs`) — must run
   on a schedule (or `--watch`) to regenerate hub notes when sidecars change.
   CLI supports `--once` and `--watch` modes. A `--watch` process is required
   for live HUD updates.
3. **Watchdog** (`scripts/orchestration/graph-color-watchdog.mjs`) — monitors
   the daemon's health and alerts if the mirror falls behind.

This is three additional long-running processes to monitor in the development
environment. Each adds a failure mode.

#### N4 — Plugin compatibility risks

Obsidian community plugins (~2,750) are not governed by a formal API stability
contract. Plugin updates can break:

- Tag namespace (folder-id mismatch incident on 2026-05-02 demonstrated this
  risk — a plugin update changed internal folder IDs, breaking graph queries).
- MCP server compatibility with Claude Code config format.
- Bases schema compatibility with typed Properties.
- Excalidraw scripting API.

Mitigation: BRAT plugin for beta-testing updates before adoption; Obsidian Git
auto-commit allows rolling back the vault if a plugin update corrupts state;
key plugins are pinned to specific versions in the vault's `.obsidian/` config.

#### N5 — Manual API key generation step

The Local REST API plugin requires a one-time manual API key generation in
Obsidian's plugin settings UI. This cannot be scripted (no CLI for plugin
settings). The key must then be copied into Claude Code's MCP config
(`.claude/mcp.json`). This is a manual step in an otherwise fully automated
pipeline. It must be repeated if the key is rotated or the vault is recreated.

#### N6 — Learning curve for Daniel as vault operator

Operating the HUD requires Daniel to be comfortable with:

- Obsidian Graph view navigation and colourGroup interpretation.
- Bases table views (filter/sort/group on typed Properties).
- Canvas Production Wall navigation.
- Running `node scripts/orchestration/hud-orchestrator.mjs --once` on demand.

This is a modest but real learning curve. Mitigation: `00-HUD-README.md`
documents every operational procedure with screenshots and copy-paste commands.

#### N7 — Vault is a second Git repository

The Obsidian Git plugin effectively makes the vault a second Git repo with its
own commit history, branching, and merge conflicts. This is separate from the
KLOEL source repo. Operations that touch both (e.g., "mirror a commit from
source repo into vault") require coordination across two Git histories.

## Alternatives Considered

### Alt 1 — Static `PRODUCTION_SCOREBOARD.md` generated daily by cron

**Proposed**: A single markdown file at the repo root, regenerated by a daily
cron job that reads PULSE + CI + coverage and writes a structured report.

**Rejected because**:

- Creates **two sources of truth**: the mirror vault (for code) and the
  scoreboard (for state). They will inevitably drift — the scoreboard says one
  thing, the vault tags say another.
- No visual layer. Daniel can't "see" state — he can only read a long markdown
  file.
- No MCP doorway. Every Claude session starts with a ~1500-token Read of a large
  file.
- Stateless between runs. The daily cron overwrites; there's no time-series,
  no delta-tracking, no "what changed since yesterday."

### Alt 2 — Linear or Jira as authoritative project state

**Proposed**: Use Linear (Daniel's current task manager) or Jira as the single
source of truth for production blockers, phase tracking, and provider health.

**Rejected because**:

- Requires **sync infrastructure**: a bidirectional sync between PULSE findings
  and Linear issues. This is a maintenance burden and a source of drift.
- Not **local-first**. Linear is a cloud service. If Linear is down, Daniel has
  no state. The vault is local-first (files on disk) + optionally synced.
- Linear issues don't compose with the existing PULSE/findings/mirror pipeline.
  Every PULSE finding would need an API call to create/update a Linear issue.
- Linear has no Graph view, no Canvas, no Mermaid, no Excalidraw. The visual
  triage surface is limited to Kanban boards and list views.

### Alt 3 — Custom dashboard webapp (React + REST API)

**Proposed**: Build a custom Next.js dashboard that reads PULSE_REPORT.md,
coverage data, and CI state via REST APIs, renders in a browser.

**Rejected because**:

- **Maintenance burden**: a custom webapp requires its own deployment,
  dependency updates, security patches, and UI iterations. This is a product in
  itself, not a tool.
- **Doesn't compose with existing pipeline**: the mirror daemon already
  maintains a vault with frontmatter tags and sidecars. A dashboard would
  duplicate this data layer.
- **Development cost**: building a full dashboard with filtering, sorting,
  ranking, time-series, and provider health widgets is ~100+ hours of work.
  Obsidian gives this for free via Bases, Graph, Canvas, and Dataview.
- **Separate deployment**: another thing to keep running, another URL to
  bookmark, another auth layer to maintain.

### Alt 4 — CLI-only state via `PULSE_REPORT.md` plus ad-hoc grep

**Proposed**: Keep PULSE as the only state surface. Claude reads
`PULSE_REPORT.md` on session start. Daniel greps for what he needs. No vault, no
HUD, no hub notes.

**Rejected because**:

- **Not visual**: Daniel cannot triage from a terminal. He needs spatial,
  colour-coded, at-a-glance information.
- **No synthesis**: PULSE_REPORT.md is a flat list of findings across 50+
  parsers. It has no ranking, no tier/phasing, no provider health, no CI status.
  Every consumer must synthesize manually.
- **No time-series**: PULSE_REPORT.md is overwritten on every run. There's no
  "yesterday vs today" comparison unless you commit it to git after every run.
- **No agent efficiency**: every Claude session starts with a ~5000-token Read
  of PULSE_REPORT.md. There's no "what are the top 3 next tasks?" shortcut.
- **No property model**: PULSE has no concept of "tier 2 file" or "phase 3
  module" — those are human interpretations layered on top. The vault gives
  these first-class typed Properties.

## Invariants

- The mirror daemon (`scripts/obsidian-mirror-daemon*.mjs`) is
  **constitution-locked**. No AI agent may edit, weaken, bypass, or replace it.
- The `scripts/pulse/no-hardcoded-reality-audit.ts` auditor is
  **constitution-locked**. Same prohibition.
- Hub notes (`<vault>/Kloel/00-HUD/00-*.md`) are **auto-generated only**.
  They carry the `<!-- AUTO-GENERATED — do not edit -->` marker. Any manual edit
  is overwritten on the next hub refresh.
- Sidecar JSON files carry versioned schemas (`kloel.*.v1`). Breaking schema
  changes require a new version (`v2`) with a migration plan, not silent
  mutation.
- All sidecar writes are **atomic** (tmp file + rename). No partial writes in
  the vault.
- The blocker-rank algorithm is canonical. If a different ranking is desired, the
  algorithm is changed (via OpenCode subagent), not the output.
- Claude/OpenCode subagents access vault state **exclusively via MCP**, not
  filesystem reads. This enforces the token economy gain and prevents bypass-
  drift.
